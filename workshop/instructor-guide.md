## 모듈별 강사 가이드

### M0 공급망 보안 개요 (30분)

**핵심 메시지**: "왜 지금 이게 중요한가"

**3분 설명 스크립트**:
> "2021년 Log4Shell 사태를 기억하시나요? 전 세계 수십억 개의 시스템에 영향을 준 취약점이었지만,
> 정작 많은 조직이 자신의 소프트웨어에 Log4j가 포함되어 있는지조차 몰랐습니다.
> SBOM이 있었다면 즉시 영향 범위를 파악하고 대응할 수 있었을 겁니다.
> 오늘 우리가 구축하는 체계는 바로 이런 상황을 예방하기 위한 것입니다."

**강의 체크포인트**:
- 수강생이 Log4Shell 사례의 핵심(SBOM 부재 → 영향 범위 파악 불가)을 이해했는가
- ISO/IEC 5230(라이선스)과 18974(보안)의 차이를 설명할 수 있는가

**예상 질문 TOP 3**:

| 질문 | 모범 답변 |
|------|---------|
| SBOM이 정확히 뭔가요? | "소프트웨어 부품 명세서(Software Bill of Materials)입니다. 우리 소프트웨어에 포함된 모든 오픈소스 컴포넌트와 버전, 라이선스를 목록화한 문서입니다." |
| 우리 회사도 반드시 해야 하나요? | "의무는 국가·업종·계약에 따라 다릅니다. 하지만 EU CRA, EO 14028 등 주요 규제가 SBOM을 요구하는 방향으로 가고 있어 조기 구축이 유리합니다." |
| 두 표준 모두 해야 하나요? | "선택입니다. 오늘 워크숍은 두 표준을 동시에 달성하도록 설계되어 있어 추가 부담이 거의 없습니다." |

**자주 막히는 지점**: 두 표준의 차이 혼동 → 화이트보드에 5230=라이선스, 18974=보안 두 단어로 구분

### M1 조직 + 정책 (1시간)

**핵심 메시지**: "담당자와 정책이 모든 것의 기반"

**3분 설명 스크립트**:
> "표준에서 가장 먼저 요구하는 것이 바로 '누가 책임지는가'입니다.
> 아무리 좋은 도구와 프로세스가 있어도, 담당자가 없으면 아무도 실행하지 않습니다.
> 오늘 M1에서는 Agent가 여러분 회사 상황에 맞는 조직 구조와 정책을 자동으로 생성해줍니다."

**강의 체크포인트**:
- `output/organization/` 폴더에 3개 파일 생성 확인
- `output/policy/` 폴더에 2개 파일 생성 확인

**예상 질문 TOP 3**:

| 질문 | 모범 답변 |
|------|---------|
| 담당자가 1명이어도 되나요? | "표준은 인원 수를 규정하지 않습니다. 1인 담당도 가능하지만 역할(정책 관리, 기술 지원, 교육)은 문서로 명확히 정의해야 합니다." |
| 정책은 얼마나 상세해야 하나요? | "핵심 원칙을 담되 실행 가능한 수준으로. 너무 상세하면 변경 관리가 어렵습니다. Agent가 생성한 초안을 기준으로 회사 실정에 맞게 조정하세요." |
| 기존 정책이 있는데 새로 만들어야 하나요? | "기존 정책에 표준 요구사항이 포함되어 있으면 그대로 활용하시고, 없는 부분만 추가하시면 됩니다." |

**자주 막히는 지점**: Agent에게 회사 정보 입력할 때 막힘 → 개발자 수, 배포 방식, 주요 언어 3가지만 입력해도 진행 가능

### M2 프로세스 (1시간)

**핵심 메시지**: "정책을 실행 가능하게 만드는 것이 프로세스"

**3분 설명 스크립트**:
> "'오픈소스 사용 전 승인을 받는다'는 정책은 있지만, 어디에 신청하고, 누가 검토하고, 며칠 안에 결과를 받는지가 명확하지 않으면 정책이 유명무실해집니다.
> M2에서는 이 실행 절차를 문서화합니다."

**강의 체크포인트**:
- `output/process/` 폴더에 4개 파일 생성 확인
- `process-diagram.md`의 Mermaid 다이어그램 렌더링 확인

**예상 질문 TOP 3**:

| 질문 | 모범 답변 |
|------|---------|
| CI/CD가 없어도 되나요? | "프로세스 자체는 CI/CD 없이도 구축 가능합니다. 수동 검토 단계로 시작하고, 이후 자동화를 추가하는 방식을 권장합니다." |
| 승인 단계가 꼭 필요한가요? | "표준은 검토 절차를 요구하지만 승인 단계의 형식은 자유입니다. 이메일 확인도 기록이 남으면 됩니다." |
| 프로세스 다이어그램 형식은 무엇인가요? | "Mermaid 형식으로 생성됩니다. Docusaurus나 GitHub에서 바로 렌더링됩니다." |

**자주 막히는 지점**: 취약점 대응 절차 작성 시 담당자 미지정 → M1에서 생성한 역할 정의 파일 참조 안내

### M3 SBOM 생성 + 분석 (1시간 30분)

**핵심 메시지**: "내 소프트웨어에 뭐가 들어있는지 처음으로 확인"

**3분 설명 스크립트**:
> "SBOM은 식품의 영양성분표와 같습니다. 우리가 매일 사용하는 소프트웨어에 어떤 오픈소스가 들어있는지,
> 오늘 처음으로 정확하게 확인하게 됩니다. CycloneDX 형식으로 생성된 SBOM은 라이선스 분석과 취약점 스캔의 입력값이 됩니다."

**강의 체크포인트**:
- `output/sbom/[project].cdx.json` 파일 생성 확인 (0KB 이상)
- `output/sbom/license-report.md` 생성 확인

**예상 질문 TOP 3**:

| 질문 | 모범 답변 |
|------|---------|
| Docker가 너무 느려요 | "이미지 pull이 느린 경우입니다. `samples/` 프로젝트로 먼저 진행하거나, 옆 분과 화면 공유로 함께 진행하세요." |
| SBOM이 비어있어요 | "패키지 매니저 파일(package.json, pom.xml 등)이 프로젝트 루트에 있는지 확인하세요. samples/ 프로젝트로 테스트해보세요." |
| Copyleft 라이선스가 많이 나왔어요 | "Copyleft 자체가 법적 위반은 아닙니다. 소스코드 공개 의무 이행 방법을 M2 배포 체크리스트에서 확인하세요." |

**자주 막히는 지점**: Docker 미설치/미실행 → 옆 수강생과 페어로 진행, 또는 강사 노트북 화면 공유

---

### M4 SBOM 관리 + 공유 (45분)

**핵심 메시지**: "만들고 끝이 아니라 관리가 시작"

**3분 설명 스크립트**:
> "SBOM은 한 번 만들면 끝이 아닙니다. 소프트웨어가 업데이트될 때마다 갱신해야 하고,
> 공급망 파트너(고객, 납품처)에게 공유할 수 있는 체계도 필요합니다.
> M4에서는 이 지속적인 관리 체계를 구축합니다."

**강의 체크포인트**:
- `output/sbom/sbom-management-plan.md` 생성 확인
- `output/sbom/sbom-sharing-template.md` 생성 확인

**예상 질문 TOP 3**:

| 질문 | 모범 답변 |
|------|---------|
| 얼마나 자주 갱신해야 하나요? | "릴리즈마다 갱신이 원칙입니다. CI/CD에 통합하면 자동으로 최신 상태를 유지할 수 있습니다." |
| 고객이 SBOM을 요청하면? | "M4에서 만든 sbom-sharing-template.md를 활용하세요. 공유 범위와 형식을 사전에 정의해두면 됩니다." |
| SBOM을 공개해야 하나요? | "의무 공개 여부는 계약과 규제에 따라 다릅니다. 내부 보관이 기본이고, 요청 시 제공하는 방식이 일반적입니다." |

**시간 조절**: 지연 시 이 모듈을 과제로 전환 가능

---

### M5 취약점 분석 (45분)

**핵심 메시지**: "이미 알려진 위험을 모르고 있을 수 있다"

**3분 설명 스크립트**:
> "NVD(미국 국가취약점 데이터베이스)에는 매년 수만 건의 새로운 CVE가 등록됩니다.
> 우리가 사용하는 오픈소스 컴포넌트 중 이미 알려진 취약점이 있는 것이 있을 수 있습니다.
> M5에서는 M3에서 만든 SBOM을 활용해 이를 자동으로 스캔하고 대응 계획을 세웁니다."

**강의 체크포인트**:
- `output/vulnerability/cve-report.md` 생성 확인
- `output/vulnerability/remediation-plan.md` 생성 확인

**예상 질문 TOP 3**:

| 질문 | 모범 답변 |
|------|---------|
| Critical CVE가 나왔어요, 어떻게 해요? | "당황하지 마세요. 리포트의 Remediation 섹션에 대응 방법이 있습니다. 오늘 당장 패치할 필요는 없고, 계획을 수립하는 것이 목표입니다." |
| CVE가 하나도 안 나왔어요 | "좋은 신호입니다. 최신 버전의 컴포넌트를 사용하거나, 샘플 프로젝트가 이미 패치된 버전일 수 있습니다." |
| CVSS 점수가 뭔가요? | "Common Vulnerability Scoring System. 0-10점 척도로 취약점 심각도를 수치화한 것입니다. 7.0 이상은 High로 우선 대응합니다." |

**자주 막히는 지점**: SBOM 없이 M5 진행 시 → M3로 돌아가 SBOM 먼저 생성

## 수강생 환경 문제 대응

| 문제 | 즉각 대응 |
|------|---------|
| Docker Desktop 실행 안 됨 | 옆 수강생과 페어로 진행 (화면 공유) |
| Claude Code 로그인 안 됨 | 웹 버전 claude.ai 임시 활용 (파일 저장은 수동) |
| 인터넷 매우 느림 | Docker 이미지 사전 pull 안내: `docker pull syft` |
| 노트북 없는 수강생 | 강사 화면 공유 또는 클라우드 환경(GitHub Codespaces) 안내 |
| Windows 환경 문제 | WSL2 + Docker Desktop 조합 확인, Git Bash 사용 안내 |
