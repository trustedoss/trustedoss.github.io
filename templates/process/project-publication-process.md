## 1. 공개 전 검토 절차

<!-- 5230 §3.5.1 - 오픈소스 커뮤니티 참여 정책 -->

사내 프로젝트를 오픈소스로 공개하기 전 아래를 확인한다:

| 확인 항목                           | 확인 방법           | 담당자      |
| ----------------------------------- | ------------------- | ----------- |
| 회사 IP(특허, 영업비밀) 포함 여부   | IP 체크리스트 작성  | 법무        |
| 제3자 라이브러리 라이선스 호환성    | SBOM 분석           | OSPM        |
| 개인정보·내부 시스템 정보 포함 여부 | 코드 리뷰           | 개발팀      |
| 보안 취약점 사전 점검               | 정적 분석 도구 실행 | 보안 담당자 |
| 공개 목적 및 전략적 타당성          | 경영진 검토         | 관리자      |

공개 검토 요청 시 제출 정보:

- 프로젝트명 및 저장소 URL (내부)
- 공개 목적 및 기대 효과
- 사용 라이선스 (선택 이유 포함)
- 제3자 라이브러리 목록

## 2. 라이선스 선택 기준

<!-- 5230 §3.5.1 -->

| 공개 목적          | 권장 라이선스   | 이유                          |
| ------------------ | --------------- | ----------------------------- |
| 최대 확산 목표     | MIT, Apache 2.0 | 상업적 사용 제한 없음         |
| 커뮤니티 기여 유도 | GPL v2/v3, AGPL | Copyleft 의무로 기여 환수     |
| 특허 보호 필요     | Apache 2.0      | 명시적 특허 허가 조항 포함    |
| 라이브러리 공개    | LGPL, MPL       | 독립 사용 가능, 약한 Copyleft |

최종 라이선스 선택은 OSPM + 법무팀 공동 결정.

## 3. 공개 준비

<!-- 5230 §3.5.1 - 공개 준비 절차 -->

공개 승인 후 저장소 준비:

- [ ] 민감 정보 제거 (API 키, 비밀번호, 내부 URL 등)
- [ ] `.gitignore` 및 `.gitattributes` 설정
- [ ] `LICENSE` 파일 추가 (선택된 라이선스 전문)
- [ ] `README.md` 작성 (설치·사용법·기여 방법 포함)
- [ ] `CONTRIBUTING.md` 작성 (기여 가이드라인, CLA 여부)
- [ ] `SECURITY.md` 작성 (취약점 신고 방법)
- [ ] 저작권 표시 확인: `Copyright (c) {연도} {회사명}`
- [ ] 커밋 이력에 민감 정보 없음 확인

## 4. 승인 결재 단계

| 단계              | 승인자    | 승인 기준                   |
| ----------------- | --------- | --------------------------- |
| 1차: 기술 검토    | 개발 리드 | 코드 품질, 보안 취약점 없음 |
| 2차: IP·법무 검토 | 법무팀    | 특허·영업비밀 침해 없음     |
| 3차: 최종 승인    | OSPM      | 라이선스 선택, 전략 적합성  |

---

## 5. 공개 후 유지관리 계획

공개 이후 아래 활동을 지속적으로 수행한다:

- **PR/Issue 검토**: {담당자}가 신규 PR 및 Issue를 {N 영업일} 이내 확인
- **취약점 대응**: 신고된 취약점은 `취약점 대응 절차` 기준으로 처리
- **릴리즈 관리**: 주요 변경 시 태그 및 릴리즈 노트 작성
- **라이선스 의무 지속 이행**: 의존성 추가 시 라이선스 호환성 재확인

관리 중단 기준: 비즈니스 목적 소멸 또는 경영진 결정 시. 중단 전 커뮤니티에 사전 고지 필요.
