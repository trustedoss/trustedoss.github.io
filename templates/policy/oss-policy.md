## 2. 오픈소스 사용 원칙

<!-- 5230 §3.1.5.1 라이선스 의무 검토, 18974 §4.1.1.1 보안 보증 -->

### 사용 전 검토 의무

오픈소스를 신규 도입하거나 기존 버전을 변경할 때 반드시 아래를 검토한다:

1. **라이선스 확인**: `output/policy/license-allowlist.md`의 허용 라이선스 목록과 대조
2. **라이선스 의무 이행 계획**: Copyleft 라이선스의 경우 소스코드 공개 의무 등 이행 방법 확정
3. **알려진 취약점 확인**: SBOM 생성 후 CVE 스캔으로 Critical/High 취약점 없음 확인
4. **담당자 승인**: `output/process/usage-approval.md` 절차에 따라 승인

### 라이선스 분류 기준

| 분류              | 예시                 | 배포 시 의무사항                 |
| ----------------- | -------------------- | -------------------------------- |
| Permissive (허용) | MIT, Apache-2.0, BSD | 저작권 표시, 라이선스 고지       |
| Weak Copyleft     | LGPL, MPL            | 수정 파일 소스코드 공개          |
| Strong Copyleft   | GPL-2.0, GPL-3.0     | 전체 소스코드 공개 (배포 시)     |
| Network Copyleft  | AGPL-3.0             | 네트워크 사용 포함 소스코드 공개 |

## 3. 프로그램 적용 범위 및 성과 메트릭

<!-- 18974 §4.1.4.1, §4.1.4.2 -->

### 성과 메트릭 (KPI)

이 프로그램이 달성해야 할 목표:

| 지표             | 목표                                                    |
| ---------------- | ------------------------------------------------------- |
| SBOM 최신성      | 모든 릴리즈에 최신 SBOM 첨부                            |
| 라이선스 준수율  | 신규 도입 컴포넌트 100% 사전 검토                       |
| CVE 스캔 주기    | 릴리즈마다 + 월 1회 정기 스캔                           |
| 취약점 대응 시간 | Critical: 1주일, High: 4주일, Medium: 1개월 이내        |
| 취약점 대응률    | 기한 내 해소 90% 이상 (미해소 시 사유·완화 계획 문서화) |
| 교육 이수율      | 오픈소스 관련 직군 연간 100% 이수                       |
| 갱신 주기        | 자체 인증 18개월마다 재확인                             |

> **참고**: 취약점 대응 기한은 OpenChain KWG 가이드 기준선입니다. 조직의 리스크 프로파일과 역량에 따라 Critical 24시간·High 1주일 같은 더 엄격한 기한을 내부 SLA로 적용할 수 있습니다.

### 프로그램 효과성 측정 및 개선

- **정기 평가 주기**: 연 1회 이상 KPI 달성도 및 프로세스 준수 현황을 평가한다.
- **평가 보고**: 평가 결과는 경영진에 보고하며, 미달 항목은 원인과 개선안을 함께 기재한다.
- **지속적 개선**: 평가 결과·내부 모범 사례·업계 동향을 바탕으로 정책과 프로세스를 갱신한다 (갱신 주기는 정책 검토 및 갱신 섹션 참조).
- **자원 충분성 검토**: 연 1회 오픈소스 관리 인력과 도구 예산의 충분성을 평가하고 필요 시 조정을 건의한다.

## 4. 보안 보증 정책

<!-- 18974 §4.1.1.1 -->

회사는 배포 소프트웨어의 알려진 취약점(CVE)을 체계적으로 식별·추적·대응한다:

- 모든 릴리즈 전 SBOM 기반 취약점 스캔 수행
- Critical/High 취약점은 배포 전 반드시 해소 또는 완화 계획 수립
- 배포 후 신규 취약점 발견 시 `output/process/vulnerability-response.md` 절차에 따라 대응
- 외부 취약점 신고 채널 운영: security@{회사도메인}

## 5. AI 생성 코드 정책

<!-- KWG 오픈소스 가이드 권장 사항 -->

GitHub Copilot, ChatGPT 등 AI 코드 생성 도구를 사용하는 경우, 생성된 코드에도 오픈소스 라이선스 의무가 적용될 수 있으므로 아래 기준을 준수한다.

### AI 생성 코드 사용 원칙

1. **검토 의무**: AI가 생성한 코드를 제품에 포함할 경우, 오픈소스 라이선스 준수 관점에서 일반 오픈소스와 동일하게 검토한다.
2. **출처 추적 불가 인식**: AI 생성 코드는 학습 데이터의 출처를 특정하기 어려우므로, 소스 코드 스캔 도구(SCANOSS 등)를 활용한 검증을 권장한다.
3. **도구별 이용약관 확인**: 사용하는 AI 코딩 도구의 이용약관에서 생성 코드의 저작권·라이선스 조건을 확인하고 준수한다.
