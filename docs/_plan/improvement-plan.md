## 1. 미션과 방향 {#mission}

## Context — 미션

TrustedOSS의 방향(사용자 확인): AI와 도구를 활용해 OpenChain 2026(ISO/IEC 5230·18974) 기업 오픈소스 관리를 **쉽고 정확하게** 달성하도록 돕는다. 여기에 더해 DevSecOps 자동화와 AI 코딩 거버넌스를 지원한다.

대상과 목표(추가 확인): 1차 대상은 **기업 오픈소스 관리를 처음 맡은 담당자**다. 이들이 사이트에 와서 관리 체계를 쉽게 자동화하게 한다. 문서는 AI 에이전트로 손쉽게 생성하고, 자동화 도구는 우리가 가이드·제공하는 대로 쉽게 구축한다. 나아가 많은 사람이 찾아오게(유입 성장) 하고, 궁극적으로 글로벌 표준이 되는 것을 지향한다.

즉 세 기둥:

1. (핵심) **표준 기반 OSS 관리 체계 구축** — AI 에이전트가 회사 맞춤 산출물을 자동 생성해 자체 인증 선언까지 도달.
2. **DevSecOps 자동화** — 정책을 CI/파이프라인에 강제.
3. **AI 코딩 거버넌스** — 정책을 AI 코딩 도구 시점에 강제.

현 사이트는 3트랙(체계구축/DevSecOps/AI코딩) + 레퍼런스로 이 방향을 이미 담고 있다. 다만 "쉽고 정확하게"가 약해지는 지점이 있다(조사 결과): 온보딩 과부하, 세 기둥 간 고립(특히 DevSecOps↔AI코딩), 중복 콘텐츠(SBOM·라이선스·CVE 반복), 05-tools 분산(상위 인덱스 부재), 검색 부재, 개념/실습/참조 혼재(Diátaxis), 표준 매핑 3중화.

> 디자인 정합(필수): 본 IA와 콘텐츠 개편은 이미 구현된 **Gemini API 문서 look&feel** 위에서 그대로 구현한다. 적용된 디자인 요소는 Material 토큰, Roboto, 라이트 코드블록과 언어 헤더, 3유형 노트(tip, info, 회색 blockquote), breadcrumb, 제목과 헤딩 복사 버튼이다. 신규 페이지와 컴포넌트(5분 빠른 시작, 경로 선택, AI 에이전트 허브, roadmap, 진행률, KWG 커버리지 매트릭스 표)도 모두 동일한 Gemini 디자인 시스템을 따른다. 메뉴 트리와 본문 구조의 시각적 결과가 Gemini 문서와 동일하게 읽혀야 한다. 디자인 토큰과 컴포넌트는 메모리 `project-landing-design`을 참조한다.

## 차별화 — OpenChain KWG와의 관계/포지셔닝

KWG도 기업 OSS 관리 가이드, 정책/프로세스 템플릿, 도구 가이드를 제공한다. 우리는 KWG에서 콘텐츠를 동기화하므로 경쟁자가 아니라 **실행·자동화 레이어**로 선다.

한 줄 포지셔닝: KWG는 "무엇을·왜"(표준, 가이드, 빈 템플릿)를 주고, TrustedOSS는 "어떻게, 자동으로"(AI가 회사 맞춤 산출물 생성, 복붙 자동화, 체험, 0에서 자체 인증까지 완주)를 준다.

차별점:

- 정책·프로세스: KWG는 빈 템플릿을 직접 채워야 하지만, 우리는 AI 에이전트가 5개 질문 후 회사 맞춤본을 자동 생성한다.
- 도구: KWG는 도구 가이드/링크, 우리는 복붙 CI 워크플로와 Rules, 원커맨드, 무API키 데모를 제공한다.
- 범위: KWG는 컴플라이언스 거버넌스, 우리는 거기에 DevSecOps 자동화와 AI 코딩 거버넌스(KWG 미커버)를 더한다.
- 완주: 0에서 자체 인증까지 단일 동선과 갭분석, 선언문, 산출물 완전성 검증.
- 대상/UX: 초심자 우선(5분 시작, 경로 선택, 진행 가시화, Gemini look&feel).

전제와 리스크:

- 보완 관계로 포지셔닝한다. CC BY 4.0 출처 표기와 KWG 연계 명시는 필수이며 권위에도 도움이 된다.
- 차별점은 실행 품질에 달려 있다. 에이전트 산출물이 KWG에 정확히 정렬(P0-0)되고 자동화가 실제로 돌아야 가치가 산다.
- "글로벌 표준"의 현실적 의미는 표준 자체가 아니라 그 표준을 달성하는 사실상의 구현 도구(de-facto toolkit)를 지향하는 것이다.

실행: 이 차별화를 `POSITIONING.md`에 명문화하고, 랜딩에 "왜 KWG와 함께 TrustedOSS인가" 섹션으로 노출한다.

### KWG 링크 정책

본문은 자립적으로 쉽게 쓰되, 표준 근거가 필요한 곳에서 KWG 원문으로 연결한다(보완 관계, 권위 신호, CC BY 4.0 출처). 남용하지 않고 일관된 세 곳에만 둔다.

1. KWG 정렬 산출물 페이지(정책, 프로세스, 조직)에 "표준 근거: KWG 해당 가이드/템플릿" 링크.
2. 표준 요구사항 한눈에(매핑)의 각 항목에서 KWG 원문으로.
3. 산출물 예시의 "KWG 템플릿 원본" 링크.

### 도구 선정 정책

기준: 오픈소스, 유지보수 상태, 표준 포맷(SPDX/CycloneDX) 지원, 그리고 우리 동선(생성에서 고지, 공유, 스캔까지)의 빈틈을 메우는가. 과도한 카탈로그화는 피하고 작업별로 "메인 1개 + 대안" 형태로 큐레이션한다. 전체 목록은 KWG tools 페이지로 링크한다. 특정 도구 종속을 피하려 표준 포맷 중심으로 서술한다.

3계층 큐레이션:

- 국제 표준이며 널리 쓰임: syft, grype, trivy, cdxgen, OSV, Dependency-Track (현행 유지).
- OpenChain/KWG 생태계: FOSSLight, SW360, FOSSology.
- 국내이며 KWG 참여사(1차 대상에 적합): onot(Kakao와 SK텔레콤 공동, SPDX에서 OSS 고지문 자동 생성), SK텔레콤 sbom-tools(다중 언어 CycloneDX SBOM 생성, 소스/Docker/바이너리/RootFS 분석).

배치:

- onot → 4. 프로세스의 배포 전 고지문 생성(distribution-checklist §3)에 "OSS 고지문 자동 생성" 도구로. SPDX 입력 기반이라 SBOM 생성 다음 단계와 자연 연결.
- sbom-tools → 5.1 SBOM 생성에 "여러 언어를 한 번에" 보완 옵션으로. 정직하게: 내부적으로 cdxgen과 syft를 래핑한 편의 도구이므로 메인은 syft로 두고 국내 친화 통합 래퍼로 소개.

## P0-0 (최우선) — KWG 템플릿 정합성: "정확하게"의 근간

직접 대조 결과, 우리가 생성하는 정책과 프로세스 문서는 OpenChain KWG **공식 템플릿과 구조가 상당히 다르다**. 우리는 KWG의 통합 정책+프로세스 템플릿을 모듈화(조직, 정책, 프로세스, 인증을 분리)하고 축약한 뒤 가치 항목(AI 코드 정책, KPI, 변경 운영)을 더한 형태다. 정책 문서 기준으로 KWG 공식 11절 중 다음이 우리 정책 문서에 없다: 정의(용어), 역할과 책임(정책 내), 사내 공개(정책 내), 외부 문의(정책 내), ISO 준수 선언(정책 내), 그리고 §4의 세부. 일부는 다른 산출물로 외부화했고 일부는 실제 누락일 수 있다.

갱신(#13 완료, 2026-06-05): 산출물 세트 전체로 직접 대조한 결과(→ `.claude/reference/kwg-coverage-matrix.md`), KWG 정책 11절과 프로세스 6대 프로세스는 모듈형 산출물(조직/정책/프로세스/SBOM/취약점/교육/인증)에 분산되어 **거의 모두 충족**된다. 위 "상당히 다르다"는 단일 `oss-policy.md` 기준의 관찰이었다. 실제 갭은 좁다: (1) 용어 정의 절 부재, (2) 사내 공개가 조건부 생성, (3) 추적성(KWG 절 번호)이 HTML 주석에만 존재. 따라서 #2(템플릿 정렬)는 대규모 재구조화가 아니라 좁은 보강과 추적성 표면화로 수행한다.

전제: KWG 원본은 `.claude/reference/kwg/content/ko/guide/templates/`에 동기화돼 있고 CC BY 4.0이라 출처 표기로 정렬 가능. `kwg-check`/`kwg-drift-checker`/`kwg-mapping.yaml`로 드리프트 추적 가능.

조치:

1. **커버리지 매트릭스 작성**: KWG 정책·프로세스 템플릿의 모든 절을, 우리 산출물의 어느 파일·어느 섹션이 충족하는지 1:1 매핑. 실제 누락(예: 용어 정의, 사내 공개, 외부 문의가 어디서도 안 다뤄지는지) 식별. (`kwg-mapping.yaml` 확장)
2. **소스 템플릿 KWG 정렬**: 에이전트 소스 템플릿(`templates/policy/oss-policy.md` 등, 그리고 04-process가 생성하는 프로세스 산출물)을 KWG 공식 절 구조에 맞춰 보강한다. 누락 절 추가, 제목·순서를 KWG에 맞춤. 우리 가치 항목은 "확장" 절로 명시 구분. CC BY 4.0 출처 표기.
3. **추적성 표면화**: 각 산출물 절에 ISO 항목 + KWG 절 번호 매핑을 가시적으로(현재 HTML 주석 → 표기 검토). "이 산출물은 KWG 템플릿 ○절 기준"을 본문 또는 헤더에 명시 → 글로벌 권위 신호.
4. **드리프트 유지**: `kwg-check`를 분기/업데이트마다 실행해 KWG 갱신을 우리 템플릿에 반영.

검증: 커버리지 매트릭스에서 KWG 모든 절이 "충족"이어야 함. `verify.sh`의 ISO 커버리지 정합 통과. 생성 산출물 vs KWG 템플릿 절 구조 diff 최소화.

## 개선 원칙

- **초심자 우선(zero-to-automated)**: 오픈소스 관리 경험 0인 담당자가 막히지 않게. 전문 용어 최소·즉시 호버 설명, "지금 무엇을 하면 되는지"가 항상 명확.
- **미션을 IA 1층에 노출**: "무엇을 이루나(표준 체계) → AI·도구로 어떻게(에이전트) → 확장(DevSecOps·AI코딩)".
- **쉽게**: 5분 빠른 시작 + 경로 선택 + 진행 가시화 + 복붙·원커맨드 자동화.
- **정확히**: 단일 출처(중복 제거) + 표준 매핑 정본 1개 + 검색.
- **Diátaxis**: 개념/실습/참조 분리(챕터 본문=실습 중심).
- **성장과 글로벌**: 검색 유입(SEO), 공유성, 영문 패리티를 기본값으로 삼는다. 글로벌 표준을 목표로 권위 신호를 갖춘다.

## 6. 본문 콘텐츠 상세 (목차 + 주요 내용 + 실습·데모 배치)

기존 내용에 얽매이지 않고 최선안으로 재설계한다. 모든 주제는 아래 3계층으로 일관 구성한다.

### 실습·데모 통합 모델 (전 페이지 공통)

1. **보기(See) — 0클릭/무설치**: 산출물 Best Practice 샘플과 무API키 인터랙티브 데모로 결과를 즉시 본다. 초심자가 "이걸 하면 뭐가 나오는지"를 먼저 확인. 각 챕터 상단에 "바로 체험" 카드(`:::tip`)로 데모와 샘플을 링크.
2. **해보기(Try) — 내 것 생성**: AI 에이전트 실행(`cd agents/...`)으로 회사 맞춤 산출물을 만들거나, 실제 도구(syft, grype 등)를 복사-붙여넣기로 돌린다. 각 챕터 "셀프스터디(실습)" 섹션에 단계, 예상 산출물, 완료 체크를 둔다.
3. **자동화(Automate) — 지속 강제**: DevSecOps와 AI코딩에서 복붙 가능한 CI 워크플로와 Rules 파일, 원커맨드로 상시 작동하게 한다.

데모 자산: 브라우저에서 도는 무API키 분석기(SBOM, 취약점, SAST, 시크릿 등)를 데모로 쓴다. 모든 데모를 모은 **"체험하기" 허브**를 시작하기에 두고, 랜딩 1차 CTA 옆 "지금 체험"으로 연결.

### 페이지별 목차와 주요 내용

**[시작하기] 5분 빠른 시작** — 목차: 이게 뭔가, 언제 쓰나, 지금 해보기. 주요: 가치 1문단, 사용 사례 3가지, "지금 체험"(데모) 버튼과 "내 것 만들기"(첫 에이전트) 버튼. 실습은 에이전트 1개 맛보기, 데모는 SBOM 분석기 즉시 체험.

**[시작하기] 내게 맞는 시작 경로** — 3카드(인증만 / 개발팀과 함께 / 이미 정책 있음). 각 카드가 맞춤 동선으로 연결. 본문 최소, 분기만.

**[시작하기] 개요** — 목차: 왜 필요한가, 두 표준 한눈에, 전체 여정과 산출물, 자체 인증이란. 주요: 실제 사고 사례 요약(상세는 공급망 보안으로), 표준 비교 요약(상세는 매핑으로), 여정 지도. 데모·실습 없음(개념).

**[시작하기] 공급망 보안 / SBOM 기본** — 배경 개념. SBOM 기본 페이지에 "SBOM 데모"(샘플 의존성에서 SBOM 미리보기)를 연결해 개념을 즉시 체감.

**[시작하기] 표준 요구사항 한눈에** — KWG와 ISO 정본 표(단일 마스터). 각 항목에서 담당 챕터, 에이전트, 산출물로 점프한다.

**[시작하기] AI 에이전트로 산출물 만들기 (허브)** — 목차: 에이전트란, 에이전트와 챕터와 산출물 매핑표, 상황별 추천, 공통 실행법. 주요: "어느 상황에 어느 에이전트"와 실행 전 공통 주의. 핵심 차별점을 한 페이지로.

**1. 환경 준비** — 목차: 이 챕터 할 일, 필요 도구, OS별 설치, 설치 확인, 저장소 클론. 실습: 설치 검증 명령 복붙. 보기: 설치 없이 먼저 보려면 무API키 데모로 안내.

**2. 조직 구성** — 목차: 이 챕터 할 일, 배경(왜 역할 정의가 표준 요구인가), KWG 역할 8종과 역량, 규모별 현실안, 실습, 완료 확인, 다음. 보기: 조직 산출물 샘플(스타트업/중견/대기업). 해보기: organization-designer 에이전트.

**3. 오픈소스 정책 (KWG 11절 정렬)** — 본문은 5절 §B의 11절 구조. 추가로 실습(policy-generator 에이전트, 5개 질문)과 완료 체크. 보기: 정책 샘플 비교 뷰. 자동화로 가는 다리: 정책을 AI코딩 Rules로 내보내기 링크.

**4. 오픈소스 프로세스 (KWG 6대 프로세스)** — 본문은 6대 프로세스 구조. 실습: process-designer. 보기: 프로세스 흐름도(Mermaid)와 샘플. 자동화: 사용 승인을 CI 게이트로(데브섹옵스 링크).

**5. 도구: SBOM과 취약점 (실습·데모 핵심 구역)**

- 5.0 통합 인덱스: 생성에서 관리, 취약점으로 이어지는 흐름과 통합 에이전트 안내.
- 5.1 SBOM 생성 — 목차: 왜 SBOM, 도구(syft, cdxgen), 생성 실습, CI 통합, 결과 읽는 법. 보기: 브라우저 SBOM 분석기(샘플 업로드에서 라이선스 분류). 해보기: syft 복붙 + sbom-guide 에이전트. 자동화: SBOM 생성 워크플로 복붙.
- 5.2 SBOM 관리 — 갱신 주기, 보관, 공급망 공유. 샘플 관리 계획.
- 5.3 취약점 분석과 대응 — 도구(grype, OSV), CVSS 분류, 대응 기한, VEX. 보기: 취약점 분석기 데모(샘플 SBOM에서 CVE). 해보기: grype 복붙 + vulnerability-analyst. 자동화: 취약점 게이트.

**6. 교육 체계** — 목차: 이 챕터, 배경, 직군별 커리큘럼, 실습, 완료. 해보기: training-manager 에이전트. 리소스 목록은 레퍼런스로 이동(개념·참조 분리).

**7. 자체 인증 선언** — 목차: 이 챕터, 갭 분석, 선언문 작성, OpenChain 등록, 갱신. 해보기: conformance-preparer. 보기: 갭 분석 샘플. 마무리에서 자동화 확장 분기.

## 실행 규약 (auto mode 자기 규율)

원칙: 나태 방지는 의지가 아니라 객관적 게이트와 독립 검증으로 강제한다.

1. 단위는 task 하나를 완료 정의(DoD)까지. 우선순위(ID) 순서로. in_progress에서 completed로는 게이트 통과 시에만.
2. 완료 정의(DoD): (a) 해당 task 고유 수용 기준 충족, (b) `cd website && npm run build` 통과(ko/en), (c) `verify.sh` 12/12, (d) UI 변경 시 헤드리스 캡처 확인, (e) KWG 관련은 커버리지 100%. 하나라도 불통이면 미완료.
3. 매 사이클 시작 시 재정렬: 계획 파일과 task 목록을 다시 읽어 결정사항(확정 라벨, KWG 포지셔닝, Gemini look&feel, 초심자 우선)을 재확인해 drift를 막는다.
4. 초심 게이트: 모든 산출물은 "처음 맡은 담당자가 막힘없이, 그리고 KWG에 정확히"를 통과해야 한다. 통과 못 하면 완료가 아니다.
5. 자가 채점 금지: 단계 종료 전 quality-gate, code-review, doc-qa 서브에이전트로 독립 교차 검증한다.
6. 정직 보고와 무묵살 절단: 실패, 스킵, 한계는 그대로 보고한다. 테스트 실패 상태로 완료 표기 금지. 막히면 in_progress 유지 후 후속 task를 만들고 표면화한다.
7. 사실 확인 선행: 단정 전에 grep과 실데이터로 재확인한다.
8. 진척 기록: 매 task 후 실행 현황을 갱신하고 `docs/_plan/improvement-plan.md`를 동기화한다.
9. 종료 조건: 전 task 완료 후 최종 end-to-end 검증(전 페이지 빌드, 내부 링크, 온보딩 클릭 테스트)과 완전성 비평 1회를 통과해야 전체 완료로 본다.
10. auto mode 운영(사용자 확정): 새 작업 브랜치에서 task별로 커밋한다. 커밋 메시지에 Co-Authored-By 등 AI 공동작성 트레일러를 넣지 않는다(전역 규칙). 마일스톤(P0-0, P0, P1 등)이 끝날 때마다 결과를 보고하고 계속 진행한다. 진행하며 새로 알게 된 사실이나 더 나은 안은 즉시 계획 파일과 `docs/_plan`에 반영해 계획을 계속 보강한다.
